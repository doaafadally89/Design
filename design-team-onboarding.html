Section 1

What is GitHub?

Think of it as Google Drive for code — a place in the cloud where your team's code is safely stored, tracked, and shared.

☁️
Cloud Storage
All code lives online, accessible from anywhere. No "which version is latest?" confusion.
📜
Version History
Every change is recorded — like Figma's version history, but for code. You can always go back.
👥
Team Collaboration
Multiple developers can work on the same project simultaneously without overwriting each other.
Repository

The Repository (Repo)

A repository is a project folder on GitHub. It contains all the code, files, images, and the complete history of every change ever made.

📦 my-project
📄 index.html
📄 styles.css
📁 components
📄 Header.tsx
📄 Button.tsx
📁 assets
🖼️ logo.png
📄 README.md
💡 Design Analogy
A repo is like a Figma project. It's the container that holds all the pages (files), components, and assets for one product or feature.
🔑 Key Point
Each project has its own repo. For example, we might have one repo for the main platform and another for a marketing site.
Branch

What is a Branch?

A branch is a separate copy of the code where you can make changes without affecting the main version. Like duplicating a Figma page to try a new layout.

💡 main is the "live" version. Developers create branches to work on features safely, then merge them back when ready. Just like duplicating a Figma page, experimenting, then moving the final design back to the main page.
The Git Journey

Clone, Pull, Push & PR

The 4 steps of working with GitHub — click each step to learn more

Clone — Copy the Project

One-time

When you join a project for the first time, you clone it — downloading the entire repository from GitHub to your computer. After that, you have a full local copy.

Figma analogy: Like downloading a shared Figma file to work locally — except Git keeps it connected so you can sync changes back.
Terminal
$ git clone github.com/team/project
Cloning into 'project'...
✓ Done! Project copied.

Pull — Get the Latest Changes

Every time

Do this every time you start working. Your teammates may have pushed new code. "Pull" downloads their latest changes so your local copy stays up to date.

Figma analogy: Like opening a shared Figma file and seeing "Updates available" — accept them to get the latest before making changes.
Terminal
$ git pull
Updating 3a7f2c1..b9e4d8f
3 files changed
✓ Up to date!

Push — Share Your Changes

After making changes locally, you first commit (save a checkpoint with a description), then push to send those changes up to GitHub so the team can see them.

Figma analogy: Like saving your Figma changes and hitting "Publish" so everyone on the team sees your latest updates.
Terminal
$ git add .
$ git commit -m "Add sidebar"
$ git push
✓ Pushed to GitHub!

Pull Request — Review & Merge

After pushing, you create a Pull Request on GitHub — a formal request saying "please review my changes and merge them into the main codebase." The team reviews, comments, and approves.

Figma analogy: Like sharing a prototype link with "Ready for review" — stakeholders comment, you iterate, and once approved the design moves to handoff.

Section 4

What are MCPs?

Model Context Protocol — plugins that connect Claude Code to external tools like Figma, Notion, and more.

Without MCPs
Claude Code can only see the files on your computer. To reference a Figma design, you'd have to describe it or paste screenshots.
Claude Code  ⟶  Local files only
With MCPs
Claude Code can directly read your Figma designs, access Notion docs, deploy to Netlify — all from within the IDE.
Claude Code  ⟶  Figma + Notion + Netlify + ...
Think of MCPs as Figma plugins — for Claude Code
Just like you install Figma plugins to add new capabilities (icons, illustrations, accessibility checks), MCPs give Claude Code the ability to connect to external services. Each MCP adds a new superpower.

Setting Up API Keys

Some projects need secret keys to connect to services like Figma, APIs, or databases. These go in a .env file.

What is a .env file?
A .env file stores secret keys and passwords that the project needs to connect to external services. It's like a keychain for the project.
Important: This file is never pushed to GitHub — it stays private on your computer. Each team member creates their own.
Manual approach
Copy .env.example → .env and fill in your keys:
$ cp .env.example .env
# Then edit .env and add your keys:

FIGMA_API_KEY=figd_xxxxx

Common Issues & Fixes
404 Not Found
Cause: Pages not enabled, or wrong file path
Fix: Check Settings → Pages is set to main. File names are case-sensitive.
Changes Not Showing
Cause: Browser cache or deploy still in progress
Fix: Hard refresh (Cmd+Shift+R) or wait 1-2 minutes.
Pushed to Wrong Branch
Cause: You're on a feature branch, not main
Fix: Tell Claude: "Switch to main and push"

Use GitHub Pages when...
• You want version history for every change
• The team is already using this GitHub repo
• You're sharing HTML prototypes or reports
• You want auto-deploy on every push
Use Netlify when...
• You need a quick preview link for stakeholders
• The viewer doesn't have GitHub access
• You're deploying a React or Next.js app
• You don't want to set up a repo first
